Arquitetura de Sistemas

A poucos anos muitas aplicações eram desenvolvidas utilizando a arquitetura cliente/servidor de duas camadas (ainda hoje, algumas implementações de sistemas seguem esta arquitetura).

As aplicações de duas camadas (cliente/servidor) eram desenvolvidas para acessar grandes bases de dados, e fundiam as regras usadas para manipular os dados com a interface do usuário na aplicação cliente. A tarefa do servidor era simplesmente processar e responder quantas requisições por dados fossem possíveis.

Aplicações de duas camadas realizam muitas das funções que um sistema atômico ( uma camada) efetua: elas apresentam uma interface para o usuário, recebem e processam entradas do usuário, efetuam o processamento requisitado, e relatam o resultado da requisição. Esta sequência de comandos pode ser repetida quantas vezes for necessária. Já que o servidor fornece apenas o acesso aos dados, o cliente usa seus recursos locais para efetuar a maior parte do processamento. A aplicação cliente deve conter informação sobre onde o dado reside e como ele está organizado no banco de dados. Uma vez que o dado é recebido, o cliente é responsável por formatar e exibí-lo para o usuário.

Uma das principais vantagens do modelo cliente/servidor era que múltiplos usuários acessavam simultaneamente os dados da mesma aplicação, atualizações de um computador estavam disponíveis instantaneamente para todos os computadores que tinham acesso ao servidor. Contudo, a medida que o número de clientes aumentava, o servidor rapidamente ficava sobrecarregado de requisições de clientes. Além disso, como muito da lógica de processamento estava presa a uma monolítica suíte de aplicações, alterações nas regras de negócio levavam a caras e demoradas alterações no código fonte. Embora a facilidade e flexibilidade de aplicações de duas camadas continuem a conduzir muitas aplicações comerciais de pequena escala, a necessidade por acesso mais rápido aos dados e por desenvolvimento mais rápido tem convencido os desenvolvedores de sistemas a procurar por uma nova forma de criação de aplicações distribuídas. Distribuindo a arquitetura dos sistemas em mais do que duas camadas.

Java

A linguagem Java em meados dos anos 90 foi utilizada como uma ferramenta que agiria do lado do cliente (applets java, por exemplo). Já no final dos anos 90, ela foi ganhando espaço como uma poderosa ferramenta para camadas intermediárias de uma aplicação. Tendo isto em mente a Sun começou a investir fortemente em soluções para camadas intermediárias (servlets, EJBs). Até que em 1999 ela lançou sua especificação J2EE, que fornecia uma visão abrangente da utilização do Java como solução para camadas intermediárias de um sistema.

O J2EE utiliza um modelo de 4 camadas para desenvolvimento web. Resumidamente: o navegador é executado no cliente exibindo HTML e opcionalmente utiliza JavaScript. A camada intermediária abrange duas camadas: a Camada de Apresentação e a Camada de Lógica de Negócio. A camada de dados gerencia a persistência dos dados em um banco de dados e, onde necessário, armazena dados legados.
